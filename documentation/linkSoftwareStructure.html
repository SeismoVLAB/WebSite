An Open-Source Finite Element Software for Meso-Scale Simulations
Software Structure

Seismo-VLAB is divided into two main parts:

  • Pre-Analysis
  • Run-Analysis

The global structure of Seismo-VLAB is represented in the figure bellow.

The Pre-Analysis is a Python interface and is in charge of the generation of the simulation files. External CAD software such as Gmsh (http://gmsh.info/), GiD (https://www.gidhome.com/), or Trelis(https://www.csimsoft.com/trelis) can be used to generate the finite element mesh. The Pre-analysis python interface can thus be used to load these files, interprets the mesh, assign the degree of freedom numbering at each node, generate restraints, constraints, and if it is requested, divides the mesh for to be used for parallel simulations. It is important to mention that the mesh data generated in this manner, needs to be written in pre-defined format, so that the interpreter provided in the Pre-Analysis transform this information into into Seismo-VLAB file(s) format. Note that any Pre-Analysis can be used as long as the files are written in Seismo-VLAB format discussed in details at Geometry Module, Solution Module, and Input/Output Module.

The Run-Analysis is the main core of this software, and it is in charge of generating the finite element model that comes from the Pre-Analysis and perform the desired analysis. This essentially generates the finite element matrices, assembles the contribution of each elements, and solves the resulting systems of equations. The solution generated in such process is then recorded in separated files whose format is discussed in Recorder. In the Run-Analysis component, the software's folder is divided in several sub-folder which contains all the objects needed to create a finite element model and then perform an analysis. This internal folder structure is deliberately chosen so that finding/modifying/adding new classes are simple for any user. These folders at the same time are divided into three categories,

  • Geometry Module: This module is explained in Geometry Module and is conformed by Node, Material, Section, Elements, Load, and Mesh classes. On this level the mesh of the finite element model is generated, in which nodes are placed in space, elements are defined through node connectivity, materials are assigned to the elements, and loads are specified on nodes and elements, respectively. The mesh encapsulates all these objects, which is finally used by the assembler object to generate the global mass, damping and stiffness matrices as well as the global force vector.
  • Solution Module: This module is explained in Solution Module and is conformed by Analysis, Algorithm, Integrator, LinearSystem, and Assembler classes. On this level an incremental analysis is performed for either static or dynamical purpose. The analysis uses the algorithm to take care of how the solution will be evolved for each time step. During each time step, the integrator combines the matrices and vectors using the assembler and gives this information to the solver. The solver finds the solution to the linear system (generated from the integrator) and returns it to the algorithm. The algorithm checks if convergence criteria are met, and continues to the next time step.
  • Input/Output Module: This module is explained in Input/Output Module and is conformed by Recorder and Parser classes. In this level the Parser reads the domain and analysis input file(s) and generates the Node, Element, Material, Load, Mesh objects to define a finite element problem. It also generates the analysis and how the finite element problem is going to be solved. The Recorders stores the solution at Node or Element obtained in the Solution Module in files specified by the user.

It can be seen that a Seismo-VLAB execution is mainly managed via input/outputs files. The python Pre-Analysis creates output files that are inputs for the Run-Analysis. In other words, the Pre-Analysis is an interface that allows to enter the FE model in a simpler manner. Therefore, the Pre-Analysis execution can be ommited if the input files for the Run-Analysis are provided in this format. The global input- and output-file structure in SVL is depicted in the Figure below.

The generated inputs/output files during the SVL workflow are located in the following folders:

  • ./Script.py : Main python file (generated by the user) employed to create the FE model. The Pre-Analysis python file is described in ~sec:Seismo-VLAB_Python}. We insist on this process can be skipped as long as the File.$.svl are written in Run-Analysis format.
  • ./Partition/File.$.svl : Set of partitioned input files generated in Pre-Analysis and written in Run-Analysis format. Note that the .$. represents th processor number, and each one of these files contain a subdomain.
  • ./Solution/Response.out : Output files specified by the user that includes Element, Node, Material, or Section response.
  • ./Paraview/Animation_PART$.n.vtk : VTK output files that can be display using Paraview. In the output file format $ represents the partition number and n the iteration or time step.
Attention
The Pre-Analysis requires the input file: name.$.svl. The file extensions .svl. The token $ represents the processor number starting from 0. The file name.$.svl contains all information regarding the mesh, analysis and outputs. The partitioned file must be self-cointained, this is Node that define an Element, as well as Material that define Element, and Load that are applied to those Node or Element must be contained in such file. The format of each file will be described for each one of the classes in Geometry Module and Solution Module, however it is recommended to generate these files using a Pre-Analysis program, for instance a high-level language such as Python (https://www.python.org/) or MATLAB (https://www.mathworks.com/products/matlab.html).
